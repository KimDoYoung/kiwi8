# kiwi8 인증(Authentication) 시스템 설계

본 문서는 kiwi8 프로젝트의 인증 시스템 현황을 분석하고, 보안성과 사용자 편의성(UX)을 강화하기 위한 향후 개선 설계를 정의합니다. 

현재 시스템은 단일 JWT(Access Token)에 의존하고 있으며, 외부 인메모리 DB(Redis)를 배제하고 내장 SQLite를 활용하여 세션을 관리하는 방향으로 설계합니다.

---

## 1. 현재 인증 구조 분석

### 1.1 현행 구현 방식
*   **방식**: JWT(JSON Web Token) 기반 단일 토큰 인증.
*   **발급**: `/login` 호출 시 `settings` 테이블의 단일 사용자 계정(`user_id`, `user_pw`)과 대조 후 Access Token 1개 발급.
*   **저장**: 클라이언트의 Cookie (`httponly=True`, `samesite='lax'`)에 저장하여 XSS 공격을 1차적으로 방어.
*   **검증**: FastAPI 미들웨어(`JWTAuthMiddleware`)에서 모든 요청에 대해 토큰 유효성을 검사.
*   **만료**: 토큰 유효기간(예: 30분)이 지나면 API 요청 시 `401 Unauthorized` 반환 -> 프론트엔드에서 로그인 모달 표시.

### 1.2 현행 구조의 문제점 (한계)
1.  **세션 연장 불가 (Refresh Token 부재)**: 사용자가 앱을 활발히 사용 중이더라도 토큰이 만료되면 무조건 로그아웃(또는 로그인 모달)이 발생하여 UX를 저해함.
2.  **토큰 탈취 시 위험**: Access Token의 유효기간을 길게 설정하면 탈취 시 위험이 커지고, 짧게 설정하면 사용자가 자주 로그인을 해야 하는 딜레마 존재.
3.  **서버 측 통제(강제 로그아웃) 불가**: JWT는 Stateless 특성을 가지므로, 한 번 발급된 토큰은 만료되기 전까지 서버에서 임의로 무효화(Revoke)할 수 없음.

---

## 2. 인증 시스템 개선 설계 (Refresh Token + SQLite 도입)

보안 강화와 사용자 편의성(Silent Refresh)을 위해 **Access Token + Refresh Token (Dual Token)** 구조를 도입합니다. Redis 대신 기존 프로젝트 스택인 **SQLite**를 세션 저장소로 활용합니다.

### 2.1 토큰 정책 (Dual Token System)

| 구분 | Access Token | Refresh Token |
| :--- | :--- | :--- |
| **목적** | API 접근 권한 증명 | 새로운 Access Token 발급용 |
| **유효기간** | 짧음 (예: 15분 ~ 30분) | 김 (예: 7일 ~ 14일) |
| **저장 위치** | Cookie (`httponly=True`, `samesite='lax'`) 또는 메모리 | Cookie (`httponly=True`, `secure=True`, `path='/api/auth/refresh'`) |
| **서버 DB 저장** | 저장하지 않음 (Stateless) | **SQLite에 해시값 저장 (Stateful)** |

### 2.2 SQLite 기반 Refresh Token 관리 테이블 설계
Redis를 사용하지 않으므로, 발급된 Refresh Token을 추적하고 폐기(Revoke)하기 위해 SQLite에 `refresh_tokens` 테이블을 신설합니다.

```sql
CREATE TABLE IF NOT EXISTS refresh_tokens (
    token_id TEXT PRIMARY KEY,             -- Refresh Token 고유 식별자 (UUID 등)
    user_id TEXT NOT NULL,                 -- 사용자 ID
    hashed_token TEXT NOT NULL,            -- Refresh Token 해시값 (보안용)
    expires_at TIMESTAMP NOT NULL,         -- 토큰 만료 일시
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_revoked INTEGER DEFAULT 0           -- 폐기 여부 (1: 폐기됨, 강제 로그아웃 시 사용)
);

CREATE INDEX IF NOT EXISTS idx_refresh_tokens_user ON refresh_tokens(user_id);
```

### 2.3 인증 프로세스 시나리오 (Flow)

#### A. 로그인 (Login)
1. 사용자가 아이디/비밀번호 입력 후 `/login` 요청.
2. 서버는 자격 증명 확인 후 **Access Token**과 **Refresh Token**을 동시에 생성.
3. Refresh Token의 메타데이터(해시값, 만료일)를 SQLite `refresh_tokens` 테이블에 저장.
4. 두 토큰을 각각의 용도에 맞게 Cookie에 담아 클라이언트에 응답.

#### B. API 요청 및 검증 (API Request)
1. 클라이언트는 API 요청 시 Cookie에 담긴 Access Token을 자동 전송.
2. 서버(`JWTAuthMiddleware`)는 Access Token의 서명과 만료 여부를 검사.
3. **정상**: API 로직 수행 후 응답.
4. **만료 (401 발생)**: 클라이언트 프론트엔드의 Axios Interceptor가 `401`을 감지.

#### C. 토큰 자동 갱신 (Silent Refresh)
1. Axios Interceptor가 401을 감지하면, 백그라운드에서 `/api/auth/refresh` API를 호출.
2. 서버는 요청 쿠키에 있는 Refresh Token을 읽음.
3. 서버 검증 단계:
   - 토큰 자체의 서명 및 만료일 확인.
   - SQLite `refresh_tokens` 테이블 조회 -> 존재 여부, 만료 여부, 폐기(`is_revoked`) 여부 확인.
4. **검증 성공**: 새로운 Access Token 발급. (보안을 위해 Refresh Token Rotation을 적용하여 Refresh Token도 새로 발급 및 DB 업데이트 권장).
5. 클라이언트는 새로 발급받은 Access Token으로 실패했던 원래 API 요청을 재시도.
6. **검증 실패 (Refresh Token 만료 또는 폐기)**: 최종적으로 로그인 모달 팝업 표시.

#### D. 로그아웃 (Logout)
1. 사용자가 명시적 로그아웃 요청 (`/logout`).
2. 서버는 전송받은 Refresh Token을 식별하여 SQLite `refresh_tokens` 테이블에서 해당 레코드를 삭제하거나 `is_revoked = 1`로 마킹.
3. 클라이언트의 Access/Refresh Token Cookie를 모두 만료(삭제) 처리.

## 3. 프론트엔드 대응 (Axios Interceptor 개선)

프론트엔드는 현재 작성된 '세션 오버레이 모달'을 최종 Fallback으로 유지하면서, 모달이 뜨기 전에 토큰을 갱신하는 로직을 추가합니다.

```javascript
// frontend/src/services/api.ts (예상 구현)
api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    
    // 401 에러이고, 아직 재시도하지 않은 요청이라면
    if (error.response?.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true;
      
      try {
        // Refresh API 호출 (쿠키에 Refresh Token이 포함되어 전송됨)
        await axios.post('/api/auth/refresh', {}, { withCredentials: true });
        
        // Refresh 성공 시 원래 요청 재시도
        return api(originalRequest);
      } catch (refreshError) {
        // Refresh Token마저 만료/유효하지 않은 경우 -> 최종 로그아웃 (모달 띄우기)
        useAuthStore.getState().expireSession();
        setStatusMessage('세션이 만료되었습니다. 다시 로그인해 주세요.', 'error');
        return Promise.reject(refreshError);
      }
    }
    
    return Promise.reject(error);
  }
)
```

## 4. 최종 구현 상세 (Implementation Details)

설계된 내용을 바탕으로 다음과 같이 실제 구현을 완료하였습니다.

### 4.1 백엔드 구현
*   **보안 강화**: Refresh Token은 `SHA-256`으로 해싱하여 DB에 저장함으로써, DB가 유출되더라도 토큰 원본을 알 수 없게 설계했습니다.
*   **쿠키 보안**: 
    *   `Access Token`: 모든 경로(`/`)에서 전송되지만 유효기간이 짧습니다.
    *   `Refresh Token`: 오직 `/kiwi8/api/auth/refresh` 경로로 요청할 때만 브라우저가 전송하도록 제한하여 보안 노출 면적을 최소화했습니다.
*   **Stateful 로그아웃**: 로그아웃 시 DB의 해당 `token_id`를 즉시 `is_revoked = 1`로 변경하여, 탈취된 토큰이더라도 즉시 무효화되도록 구현했습니다.

### 4.2 프론트엔드 구현
*   **Silent Refresh**: 사용자가 인지하지 못하는 사이에 401 에러를 가로채 토큰을 갱신하고, 실패했던 원래의 요청을 재시도(Retry)하여 작업 흐름을 유지합니다.
*   **최종 보호막**: Refresh Token까지 만료된 최악의 경우에만 이전에 구현한 '세션 오버레이 모달'을 띄워 사용자가 작성 중인 데이터를 보호합니다.

## 5. 수정 및 생성된 파일

이번 인증 시스템 고도화로 인해 변경된 파일 목록입니다.

### 5.1 백엔드 (Python/FastAPI)
*   **`backend/core/config.py`**: Refresh Token 만료 시간 및 쿠키 이름 설정 추가.
*   **`backend/core/security.py`**: JWT Refresh Token 생성 함수 추가 및 만료 시간 처리 개선.
*   **`backend/domains/services/auth_service.py` (신규)**: SQLite 기반 `refresh_tokens` 테이블 CRUD 로직 구현.
*   **`backend/domains/services/dependency.py`**: `AuthService`를 시스템 의존성 주입 목록에 등록.
*   **`backend/api/v1/endpoints/home_routes.py`**:
    *   `POST /login`: Dual Token 발급 및 DB 저장 로직 구현.
    *   `GET /logout`: DB 내 토큰 폐기 및 쿠키 삭제 로직 구현.
    *   `POST /api/auth/refresh` (신규): 토큰 재발급 엔드포인트 구현.
*   **`sqls/kiwi8_ddl.sql`**: `refresh_tokens` 테이블 및 인덱스 정의 추가.

### 5.2 프론트엔드 (React/TypeScript)
*   **`frontend/src/services/api.ts`**: Axios Interceptor에 Silent Refresh 및 자동 재시도 로직 구현.
*   **`frontend/src/store/authStore.ts`**: 세션 만료 상태 관리를 위한 `expireSession` 액션 추가.

## 6. 결론 및 향후 과제
본 구현을 통해 Redis 없이도 안전하고 UX가 뛰어난 인증 시스템이 완성되었습니다. 향후 **Refresh Token Rotation (RTR)** 로직을 강화하여, 토큰 갱신 시마다 Refresh Token도 새로 발급받게 하면 더욱 완벽한 보안 체계를 갖출 수 있습니다.
